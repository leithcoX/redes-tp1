\section{Actividades}\label{sec:actividades} % (fold)
\begin{problem}
    Se vinculan los siguientes conceptos:
	\begin{enumerate}
        \item \textbf{gRPC}: Marco de trabajo para implementar llamadas  a procedimientos remotos.
        \item \textbf{ HTTP/2}: Protocolo de aplicación que permite  multiplexación de múltiples flujos sobre una  única conexión.
        \item \textbf{Transmission Control Protocol (TCP)}: Protocolo de transporte confiable y  full-duplex sobre el cual se establece la  conexión.
        \item \textbf{Stream}: Flujo independiente de datos dentro de una  conexión multiplexada.
        \item \textbf{Channel}: Conexión lógica que agrupa múltiples  llamadas RPC sobre una misma conexión física.
        \item \textbf{Protocol Buffers}: Serialización binaria compacta definida  mediante archivos {\color{ForestGreen}\texttt{.proto}}
        \item \textbf{StatusCode}: Código que indica éxito o tipo de error en una  llamada remota.
        \item \textbf{Unary RPC}: Tipo de llamada que envía un único request  y recibe una única respuesta.
        \item \textbf{Bidirectional Streaming RPC}:  Tipo de llamada donde cliente y servidor  pueden enviarse múltiples mensajes sin orden  predefinido.
        \item \textbf{Deadline}: Límite de tiempo máximo permitido para que  una llamada RPC finalice.

	\end{enumerate}
\end{problem}

    En loopback, dado que es la misma máquina, no hay casi retardos de propagación, y el procesamiento es la única causa de la demora.

    Por otro lado, en la red física, los paquetes que salen de la estación hacia el router, deben pasar a través de un adaptador USB, y luego pasar a través de otro adaptador USB hacia la estación destino.
    Además de ser un camino más largo, también está el tráfico de la red, si hay mucho tráfico, incrementa el retardo de propagación de los paquetes.
    Finalmente también agrega el procesamiento de la aplicación.

        \item Para ejecutar las 100 consultas, se necesitó un socket de escucha en el servidor para aceptar conexiones, un socket del cliente para realizar las peticiones, y un socket en el servidor para enviar los datos.
            Puede decirse que las transacciones siguen un patrón concurrente ya que HTTP/2 garantiza concurrencia gracias a la multiplexación de streams.

            \texttt {
            --- Estadísticas RTT ---
            \\ Cantidad total de requests: 100
            \\ Errores: 0
            \\ RTT medio (exitosos): 105.810 ms
            \\ Desvío estándar: 3.503 ms
            }

            El desvío estándar aumentó hasta 3.5\,ms pese a estar en una misma máquina.
            Aúnque el delay jitter está configurado en 0, tanto el delay inducido como factores externos a la aplicación pueden variar el desvío de los datos, por lo que es consistente con el valor nulo del delay jitter.
            Las respuestas tienden a estar ordenadas.
            En pocos paquetes ocurre que una petición con un stream ID correspondiente mayor que otro se atienda después.

            Los 100 milisegundos de delay, se reflejan en la reducción de segmentos TCP enviados, ya que validaron paquetes con menos frecuencia.
            Existe un RTT no despreciable, ya que al ser un protocolo fiable, TCP debe ordenar los segmentos que lleguen, y además realizar las validaciones correspondientes para cada uno.

            Otro contexto donde se percibe delay en la internet global, es cuando hay mucho tráfico en la misma red, o cuando se caen muchos enlaces por fallas en el servicio.
